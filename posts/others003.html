🔧 .dll 是什麼檔案？帶你了解 GCC 編譯器運作原理！

張家銘 · 2025/08/16

⚙️ GCC 編譯流程

GCC (GNU Compiler Collection) 並不是單純「把 C 原始碼轉成可執行檔」的一個單一工具，而是一整套工具鏈 (toolchain)。雖然我們表面上只看到它能把 .c 檔轉成 .exe（Windows）或 a.out（Linux），但實際上它的中間過程分成好幾層：

  1. Preprocessing (預處理)：展開 #define、移除註解、展開 #include
    （這個階段只會把標頭檔的宣告展開到程式碼裡，並不包含方法的定義！）
  2. Compilation (編譯)：把 C/C++ 原始碼轉換成組合語言 (.s)。
  3. Assembly (組譯)：把組合語言轉成"可重定位目標檔" (.o)，雖然這裡面包含了機器碼他，但它還不是最終的執行檔，因為它還沒有把函式庫給連結進來。它裡面會包含符號表 + 重定位表，用於在linker層把定義加進來!
  4. Linking (連結)：把所有 .o 檔與需要的函式庫整合，生成最終的可執行檔。

不過有趣的是，我們可以手動停在某個階段來查看中間產物：


# 顯示 GCC 中間產物
$ gcc -E hello.c -o hello.i   # 預處理 → 產生 .i
$ gcc -S hello.i -o hello.s   # 編譯 → 產生 .s (組合語言)
$ gcc -c hello.s -o hello.o   # 組譯 → 產生 .o (機器碼/目標檔)
$ gcc hello.o -o hello        # 連結 → 產生可執行檔

🔧 動態連結？那是什麼？

問題來了：如果很多程式都用到 stdio，是不是每個可執行檔裡都要複製一份？這樣太沒效率了。於是就有了動態連結！

所謂動態連結，就是在「連結階段」不把函式的機器碼直接複製進程式，而是只放一個引用 (reference)。等到程式執行時，才會去 .so 或 .dll 裡面載入真正的定義。

📦 建立共享函式庫 (.so / .dll)

如果想讓多個程式共用同一份程式碼，就可以建立「共享函式庫」。在 Linux 是 .so，在 Windows 是 .dll。

// mathlib.cpp
extern "C" int add(int a, int b) {
    return a + b;
}

# Linux: 產生 .so
$ g++ -fPIC -shared mathlib.cpp -o libmath.so

# Windows (MinGW): 產生 .dll
$ g++ -shared mathlib.cpp -o mathlib.dll

在另一個程式中，就能直接使用：

// main.cpp
#include <iostream>
extern "C" int add(int, int);

int main() {
    std::cout << add(2, 3) << std::endl;
    return 0;
}

$ g++ main.cpp -L. -lmath -o main
$ ./main  # 輸出 5

小提醒：在 Windows 上要連結 .dll，通常還需要一個 .lib 檔（匯入庫），它提供「函式位置的索引」。例如：

cl /LD mylib.c   # 生成 mylib.dll + mylib.lib
cl main.c mylib.lib  # 使用匯入庫進行連結

🔗 動態連結 vs 靜態連結

連結方式會影響程式的體積、效能與維護性：

  • 靜態連結 (.a / .lib)：把函式庫程式碼直接複製到可執行檔 → 執行檔較大，但不依賴外部檔案。由於少了載入動作，效能可能稍快。
  • 動態連結 (.so / .dll)：只保留引用，程式啟動時再載入 → 執行檔體積小、可共用、更新方便，但需要確保系統有對應的函式庫。

📊 優劣勢比較

類型優點缺點
靜態連結不依賴外部庫、部署簡單檔案大、更新困難（修改函式需重新編譯整個程式）
動態連結檔案小、可共用、更新容易版本不一致可能出錯、效能略低

🌍 跨語言整合

更有趣的是，GCC 產生的 .so / .dll 還能被 Python、Java、C# 等語言呼叫。例如 Python 可以用 ctypes：

import ctypes
lib = ctypes.CDLL("./libmath.so")
print(lib.add(10, 20))  # 輸出 30

這也是為什麼理解「編譯器 + 連結」的原理，可以讓你更靈活地整合不同語言與系統。
